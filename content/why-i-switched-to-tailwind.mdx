---
title: Why I switched to Tailwind CSS
date: "2026-06-10"
description: Utility-first CSS felt wrong until it felt completely right.
---

I resisted Tailwind for a long time. The class strings looked like noise. I was convinced that separating concerns — HTML structure here, CSS rules there — was non-negotiable. I was wrong, in the specific way that you're wrong when you're confusing familiarity with correctness.

## What I was doing before

The classic approach: a stylesheet full of BEM classes, carefully namespaced, growing without bound.

```css
/* Six months into a project */
.card { ... }
.card__header { ... }
.card__header--featured { ... }
.card__header--featured-dark { ... }
.card__header--featured-dark-compact { ... }
```

Every new variant is a new rule. Every new rule is more specificity to manage. The stylesheet becomes a negotiation between past-you who wrote the rules and present-you who needs to break them.

## The thing Tailwind actually solves

The real problem with CSS-in-stylesheets isn't the syntax. It's that CSS is global and append-only. You add new rules but rarely delete old ones because you can't be sure something isn't relying on them.

Tailwind sidesteps this entirely. The "stylesheet" is generated from exactly the classes you use in your markup — nothing more. Delete a component, and its styles disappear too.

```tsx
// Before: you have to remember to delete the CSS when this component goes away
<div className="featured-card featured-card--compact">

// After: the styles live here, and they go away when this does
<div className="rounded-xl border border-border bg-card p-4 shadow-sm">
```

## Responsive and state variants

Where Tailwind earns its keep most is responsive design. Instead of media queries scattered across a stylesheet, breakpoint prefixes sit right on the element:

```tsx
<div className="
  grid
  grid-cols-1
  sm:grid-cols-2
  lg:grid-cols-3
  gap-4
  sm:gap-6
">
```

Same for hover, focus, and dark mode states. Everything that applies to an element is readable in one place.

## The design constraint that's actually a feature

Tailwind's default scale — spacing in multiples of 4px, a curated type scale, a limited color palette — forces consistency. When all your spacing comes from the same scale, things align. When all your colors come from the same palette, things harmonize. This isn't a limitation; it's the point.

```tsx
// Every spacing value is from the same scale
// No more margin: 13px because it "looked right"
<div className="p-6 mt-4 mb-8 gap-3">
```

## What I actually gave up

In practice: not much. The main thing I lost was the ability to write complex descendant selectors easily (`nav > ul > li > a`). In return I gained a codebase where I can understand any component's styles without leaving the component file.

The class strings are verbose. I don't enjoy reading `rounded-xl border border-gray-200 bg-white p-6 shadow-sm hover:shadow-md transition-shadow` in a diff. But I enjoy it more than I enjoyed hunting through stylesheets for the rule that was overriding my margin.
